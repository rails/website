# Agent: jekyll-validator

## Purpose
Validate all Jekyll Liquid HTML files generated by `section-implementer` against Jekyll/Liquid framework rules before visual QA runs. Catches framework-level errors that won't surface as visual differences but will cause build failures, data access errors, or silent rendering bugs.

## When it runs
After `section-implementer`, before `visual-qa`. If issues are found, they are returned directly to `section-implementer` for a targeted fix — this does not count as a visual QA iteration.

## Inputs
- `files_written`: list of files created or modified by `section-implementer`

## Steps

### 1. Read all generated files
Use the Read tool to load every `.html` file from `files_written`.

### 2. Run the checklist

For each file, check every rule in the sections below. Flag violations with:
- File path and line number
- The rule violated
- The exact fix required

---

## Jekyll front matter rules (`.html` pages)

### Front matter
- Must open and close with `---` fences on page files in `world/2026/`. Includes (`_includes/`) do NOT use front matter.
- Front matter values must be valid YAML: no tabs, valid key/value pairs, strings with special characters quoted.
- Never add front matter to `_includes/` files — they are partials, not pages.

---

## Liquid template rules

### Tag syntax
- Every block tag must be properly closed:
  - `{% if ... %}` → `{% endif %}`
  - `{% for ... %}` → `{% endfor %}`
  - `{% unless ... %}` → `{% endunless %}`
  - `{% capture ... %}` → `{% endcapture %}`
  - `{% comment %}` → `{% endcomment %}`
- Flag any unclosed tag as a hard error — it will cause a build failure.

```liquid
<!-- WRONG — missing endfor -->
{% for speaker in site.data.world['2026'].speakers %}
  <div>{{ speaker.name }}</div>

<!-- CORRECT -->
{% for speaker in site.data.world['2026'].speakers %}
  <div>{{ speaker.name }}</div>
{% endfor %}
```

### Variable output syntax
- Variables are output with `{{ variable }}` — double curly braces.
- Filters are applied with `|`: `{{ speaker.name | upcase }}`.
- Never use JavaScript-style expressions inside `{{ }}`.

### Include syntax
Correct include syntax (no commas, no parentheses):
```liquid
{% include world/2026/components/speaker_card.html name=speaker.name title=speaker.title %}
```

**WRONG:**
```liquid
{% include world/2026/components/speaker_card.html name=speaker.name, title=speaker.title %}
{% include world/2026/components/speaker_card.html(name=speaker.name) %}
{% include "world/2026/components/speaker_card.html" name=speaker.name %}
```

Flag any include with commas between parameters, parentheses, or quoted filename (unless the filename contains a variable — `{% include {{ variable }} %}` is valid but rare).

### Variables in partials (includes)
**All variables passed via include MUST be accessed with the `include.` prefix inside the partial.**

```liquid
<!-- WRONG — in _includes/world/2026/components/speaker_card.html -->
<h3>{{ name }}</h3>
<p>{{ title }}</p>

<!-- CORRECT -->
<h3>{{ include.name }}</h3>
<p>{{ include.title }}</p>
```

Flag any `{{ variable_name }}` inside an `_includes/` file that does not use `include.` prefix, unless the variable is defined by an `{% assign %}` tag within the same file, or it's a Jekyll global (`site`, `page`, `layout`).

### Accessing data files with numeric year key
Data files under `_data/world/2026/` must be accessed with the year in quotes:
```liquid
<!-- CORRECT -->
{{ site.data.world['2026'].speakers }}

<!-- WRONG — will fail silently -->
{{ site.data.world.2026.speakers }}
```

Flag any data access pattern using `site.data.world.2026` without quotes.

### Loops
Use Liquid `{% for %}` loops, never JavaScript array methods:
```liquid
<!-- CORRECT -->
{% for speaker in site.data.world['2026'].speakers %}
  <div>{{ speaker.name }}</div>
{% endfor %}

<!-- WRONG -->
{{ speakers.map(s => s.name) }}
```

### Conditionals
Use Liquid `{% if %}` tags, never JavaScript ternaries:
```liquid
<!-- CORRECT -->
{% if speaker.featured %}
  <span class="featured-badge">Featured</span>
{% endif %}

<!-- WRONG -->
{{ speaker.featured ? '<span>Featured</span>' : '' }}
```

### Assign variables
Local variables must use `{% assign %}`:
```liquid
{% assign featured_speakers = site.data.world['2026'].speakers | where: "featured", true %}
```

## JavaScript rules

### DOMContentLoaded wrapper
All JavaScript must be wrapped inside a `DOMContentLoaded` listener:
```html
<script>
document.addEventListener('DOMContentLoaded', function() {
  // all JS here
});
</script>
```

Flag any `<script>` tag that directly calls `document.querySelector` or other DOM methods outside a `DOMContentLoaded` callback (or equivalent — `window.onload` is also acceptable but not preferred).

### No framework JS
Flag any use of React, Vue, Angular APIs, `useState`, `useEffect`, `ReactDOM.render`, JSX syntax, or TypeScript type annotations inside `<script>` tags.

### No `"use client"` directive
This is a Next.js/React Server Components convention and has no meaning in Jekyll. Flag it if found.

---

## No Astro/React/TypeScript artifacts

Flag any of the following as hard errors:
- `Astro.props` — not valid in Jekyll
- `interface Props {` — TypeScript; not valid in Liquid templates
- `client:load`, `client:visible`, `client:media` — Astro hydration directives
- `"use client"` — Next.js directive
- `useState(`, `useEffect(`, `useRef(` — React hooks
- `import React from`, `import { useState }` — React imports
- `.astro` file extension references in include paths
- `.tsx` file extension references in include paths

---

## Build verification

After checking all rules, run the build to confirm no compile errors:
```bash
bundle exec jekyll build 2>&1 | tail -30
```

A clean build is a hard requirement — do not pass validation if the build fails.

---

## Output

### If all rules pass and build succeeds:
```
✓ Jekyll validation — PASS
  Files checked: <list>
  Build: clean
```
Proceed to `visual-qa`.

### If violations found:
```
✗ Jekyll validation — FAIL
  Return to section-implementer with the following fixes:

  _includes/world/2026/components/speaker_card.html, line 5:
  → RULE: Variables in partials require include. prefix
  → FIX: Change {{ name }} to {{ include.name }}

  _includes/world/2026/homepage_sections/speakers.html, line 12:
  → RULE: Data access requires quoted year key
  → FIX: Change site.data.world.2026.speakers to site.data.world['2026'].speakers

  _includes/world/2026/homepage_sections/schedule.html, line 18:
  → RULE: Unclosed for tag
  → FIX: Add {% endfor %} after the closing </div> on line 22
```

Send the fix list back to `section-implementer`. The implementer applies only the listed fixes and returns. Re-run `jekyll-validator` until PASS — this loop does **not** count toward the 3-iteration visual QA limit.
